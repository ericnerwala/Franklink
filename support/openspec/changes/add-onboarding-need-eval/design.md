## Context
Onboarding currently ends at value acceptance. We need to (1) persist a concise value summary to a new DB column, and (2) add a structured needs-intake loop before marking onboarding complete.

## Goals / Non-Goals
- Goals: capture a 2-3 line value summary in `all_value`, run a needs-eval loop (ask/accept only) for 2-4 turns, capture a 2-3 line needs summary in `all_demand`, and complete onboarding afterward.
- Non-Goals: integrating Zep into summarizers, writing to `users.needs`, changing recommendation/networking behavior.

## Decisions
- Add `needs_eval` as a new onboarding stage.
- Value acceptance triggers summarization and immediately asks the first needs question in the same response (2-3 lines total).
- Needs loop uses a parallel state block `personal_facts["frank_need_eval"]` and an LLM evaluator with only ask/accept outcomes.
- Summaries are generated by separate LLM calls, fed only with user profile + personal_facts + turn_history (no Zep).
- Onboarding is only marked complete after needs acceptance + `all_demand` stored.

## Risks / Trade-offs
- Summary LLM failure: mitigate with a deterministic fallback built from `user_value`/`user_need` or the last user message.
- Accept response constraints: value-eval LLM outputs no question marks on accept; we must compose the first needs question in code.

## Migration Plan
1) Add `all_value` and `all_demand` text columns to `users`.
2) Update data models and profile hydration to include new columns.
3) Add needs-eval prompt + evaluator + node.
4) Update onboarding routing and status handling for `needs_eval`.
5) Deploy with logs enabled for needs loop and summary storage.
